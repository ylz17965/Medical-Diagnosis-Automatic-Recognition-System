## AI在作品中的应用说明

本作品在以下4个核心场景中应用了AI技术：

### 1. 智能问诊对话

基于通义千问大语言模型，结合自主研发的**14种用户意图识别器**与**10种对话状态机**，实现类医生的多轮问诊流程。系统能够从模糊主诉开始，主动追问症状细节（部位、持续时间、严重程度、伴随症状），逐步收集完整病史信息后给出个性化的诊断建议与就医指导。支持流式SSE输出，实现打字机效果的实时对话体验。

### 2. RAG检索增强问答

用户提交问题后，系统执行以下AI流水线：
- 调用 `text-embedding-v3` 将问题向量化
- 在 PostgreSQL pgvector 扩展中进行近似最近邻搜索（HNSW索引加速）
- 同步查询医学知识图谱获取结构化实体关系
- 使用 `qwen3-rerank` 对混合检索结果进行精排
- 将精选上下文注入LLM生成带来源引用和置信度评分的回答

**这一机制确保了每个AI回答都有据可查、有源可溯。**

### 3. 医学影像三维可视化

使用 VTK.js 在浏览器端实现肺部CT影像的体渲染，用户无需安装专业医学软件即可在网页中交互式查看三维肺部结构。系统预置了6种临床常用窗位预设（肺窗、仅肺部、肺部+血管、纵隔窗、骨窗），一键切换即可突出显示不同的解剖结构。支持 DICOM 和 MHD/RAW 两种医学影像格式的拖拽加载。

### 4. 可解释AI与可信机制

每个AI回答附带三重可信保障：
- **来源追溯**：标注回答中每条医学信息的来源（知识图谱/医学文献/临床指南）
- **置信度评分**：综合来源可靠性、检索相关性、多源一致性计算置信度（0-100%），低于50%自动提示"建议就医"
- **推理链展示**：展示从症状到疾病到治疗建议的完整推理路径

## 作品简介

本项目是一个面向肺部疾病领域的**AI智能问诊与CT三维可视化辅助系统**，旨在解决当前AI医疗助手普遍存在的"回答缺乏依据、无法多轮问诊、黑盒不可解释"三大痛点。

### 系统架构

系统采用前后端分离架构：
- **前端**：Vue 3 + TypeScript + VTK.js + Three.js + ONNX Runtime Web
- **后端**：Fastify + TypeScript + Prisma + PostgreSQL(pgvector) + Redis
- **AI层**：阿里云通义千问多模型协同 + LangChain 编排
- **部署**：Docker Compose 一键部署

### 核心功能

| 功能模块 | 说明 | 涉及AI技术 |
|---|---|---|
| 智能问诊 | 多轮对话式AI问诊，支持症状自查、疾病咨询 | LLM + 对话状态机 |
| CT三维可视化 | 浏览器端肺部CT体渲染，多窗位预设切换 | VTK.js 体渲染 |
| RAG知识库问答 | 向量检索+知识图谱混合搜索，回答有据可查 | Embedding + Rerank + KG |
| 肺部分割 | 浏览器端U-Net模型推理分割 | ONNX Runtime Web |

### 技术亮点

- **混合检索技术**：向量相似度搜索与医学知识图谱结构化检索的权重融合，兼顾语义理解与精准匹配
- **可信AI机制**：来源追溯+置信度评分+推理链展示，让AI回答"有据可查"
- **浏览器端3D医学影像**：无需安装专业软件，Web端实现DICOM格式CT数据的实时体渲染
- **多轮对话状态机**：14种意图识别+10种状态管理，模拟真实医生问诊流程
- **流式响应体验**：SSE流式输出+打字机效果，首字响应<2秒
- **优雅降级**：无API Key时以演示模式运行，Redis断连时自动降级

## 作品安装说明

本作品提供 **Docker 一键部署** 方式，仅需安装 Docker Desktop 即可快速启动完整系统，无需手动配置数据库和运行时环境。

### 环境要求

| 组件 | 要求 |
|---|---|
| Docker Desktop | 最新稳定版（Windows/Mac/Linux） |
| 系统内存 | 建议 ≥ 4GB 可用内存 |
| 硬盘空间 | ≥ 2GB |
| AI API Key | 阿里云通义千问 API Key（可选，无Key时以演示模式运行） |

### 安装步骤

```bash
# 1. 克隆代码仓库
git clone <仓库地址>
cd Medical-Diagnosis-Automatic-Recognition-System

# 2. 配置环境变量
#    参考 server/.env.example，编辑 server/.env.docker 文件
#    核心配置项：
#    - QWEN_API_KEY=sk-你的阿里云通义千问APIKey
#    - JWT_SECRET=自定义64位随机字符串
#    - COOKIE_SECRET=自定义64位随机字符串

# 3. 一键启动所有服务（PostgreSQL + Redis + 后端 + 前端）
docker compose up -d

# 4. 等待服务就绪（首次启动约1-2分钟）
docker compose ps  # 查看所有服务状态
```

## 设计思路

### 设计背景

当前AI医疗助手普遍存在三大核心痛点：

| 痛点 | 典型表现 | 用户影响 |
|---|---|---|
| **回答缺乏依据** | AI给出建议但用户不知道信息来源和可信度 | 用户不敢采纳建议 |
| **无法多轮问诊** | AI只做单次回答，不会像医生一样追问细节 | 诊断过于笼统、不精准 |
| **黑盒不可解释** | 用户不知道AI如何得出结论 | 降低用户信任度 |

### 设计理念

本作品以 **"问得清、看得见、信得过"** 为核心设计理念：

```
问得清 —— 多轮对话状态机，像医生一样层层追问
看得见 —— RAG来源追溯 + 知识图谱路径 + CT三维展示
信得过 —— 置信度评分 + 权威标注 + 低置信度就医提醒
```

### 技术架构设计

系统采用**前后端分离 + 微服务化 + AI多模型协同**的架构：

```
用户层（Vue 3 SPA）
    ↕ HTTP/SSE
API网关层（Nginx反向代理）
    ↕
业务服务层（Fastify）
    ├── 用户认证模块（JWT + bcrypt）
    ├── 对话管理模块（状态机 + SSE流式输出）
    ├── RAG检索模块（pgvector向量搜索 + 知识图谱）
    ├── 知识图谱模块（实体关系推理）
    ├── 影像处理模块（DICOM解析 + 体渲染数据准备）
    └── 联邦学习模块（隐私保护跨医院搜索）
    ↕
数据层（PostgreSQL + pgvector + Redis）
    ↕
AI层（阿里云通义千问多模型协同）
```

### 数据流设计

以一次典型的"症状自查"为例：

1. 用户输入"咳嗽两周，有点发热"
2. 意图识别器判定为"症状咨询"，状态机进入"症状收集"阶段
3. 系统调用 `text-embedding-v3` 对问题向量化
4. 向量在 pgvector 中检索相似医学文档（HNSW索引加速）
5. 知识图谱同步检索"咳嗽"和"发热"相关疾病节点
6. 混合结果经 `qwen3-rerank` 重排序
7. 精选上下文 + 系统提示词注入 `qwen-max` 生成回答
8. 回答附带来源引用、置信度评分、推理路径
9. SSE流式返回前端，打字机效果展示
10. 系统主动追问"咳痰吗？痰是什么颜色？"，继续多轮问诊

## 设计重点难点

### 1. RAG与知识图谱的混合检索融合

**难点**：向量检索擅长语义相似度但缺乏结构化推理能力，知识图谱擅长精确关系查询但覆盖有限。如何将两者有机融合是一大挑战。

**解决方案**：
- 设计可配置的权重融合机制（默认向量0.6 : 图谱0.4）
- 引入重排序模型对混合结果进行统一精排
- 实现去重算法消除向量检索和图谱检索的冗余结果
- 支持动态权重调整，根据查询类型自动分配最优权重

### 2. 可信可解释AI机制

**难点**：大语言模型的"幻觉"问题在医疗领域尤为致命。如何让AI每个回答都附带可信度评估和来源追溯，技术实现复杂度高。

**解决方案**：
- 构建多维度置信度评估模型：来源可靠性（0.6-0.95）+ 检索相关性 + 多源一致性 + 信息覆盖度
- 实现推理链可视化：从用户问题→检索策略→信息来源→推理步骤→最终结论的完整链路展示
- 构建权威医学文献数据库（12篇核心文献、临床指南等）
- 置信度<50%时自动触发"建议就医"安全提示

### 3. 浏览器端3D医学影像体渲染

**难点**：肺部CT扫描通常包含100-500层切片，数据量可达数百MB。在浏览器中用WebGL实现实时体渲染需要精细的性能优化。

**解决方案**：
- 使用 VTK.js 的 GPU 加速体渲染管线
- 设计6种临床窗位预设，通过传输函数（Transfer Function）一键切换
- 支持 DICOM 多文件序列和 MHD/RAW 格式
- 实现交互式旋转/缩放/平移控制
- 内置FPS监控面板辅助性能调优

### 4. 多轮对话状态机设计

**难点**：医疗问诊需要像真实医生一样逐步收集信息，状态转换逻辑复杂，需要覆盖各种用户输入场景。

**解决方案**：
- 设计14种细粒度用户意图分类（症状描述、用药咨询、检查咨询、确认、否定等）
- 实现10种对话状态（初始、症状收集、诊断、治疗咨询、用药咨询、检查咨询、澄清、确认、完成、错误）
- 状态转换表驱动，支持上下文记忆（14轮对话上下文保留）
- 紧急关键词检测（胸痛、呼吸困难等自动触发120提示）
- 追问去重机制，避免重复询问相同问题

### 5. 流式SSE响应体验优化

**难点**：大模型生成完整回答需要数秒，如何在前端实现流畅的"打字机"效果，同时支持Markdown渲染和来源卡片展示。

**解决方案**：
- 基于 NDJSON 格式的 Server-Sent Events（SSE）流式传输
- 前端逐块解析并增量渲染 Markdown 内容
- 来源引用采用占位符机制，待完整回答生成后统一渲染来源卡片
- 首字响应时间控制在2秒以内

### 6. 浏览器端ONNX肺部分割推理

**难点**：在浏览器中运行深度学习模型（U-Net）涉及WebGPU/WASM兼容性、推理速度和内存管理等问题。

**解决方案**：
- 使用 ONNX Runtime Web 作为推理引擎
- 优先使用 WebGPU 加速（Chrome 113+），自动降级到 WASM 多线程
- U-Net 模型轻量化（约29MB，ResNet34 编码器）
- 切片级推理（512×512），逐层处理避免显存溢出

## 其他说明

### 1. 演示模式降级策略

系统内置演示模式，用户未配置阿里云通义千问 API Key 时，系统自动以演示模式运行，使用预设回答模板模拟AI对话，保证评委和参观者能够完整体验系统功能和交互流程。

### 2. 优雅降级设计

- **Redis 断连**：系统自动降级运行，RAG缓存和会话存储功能失效，但核心问诊功能不受影响
- **数据库异常**：系统返回友好的错误提示，不暴露后端敏感信息

### 3. 医疗免责声明

**本系统的AI生成内容仅供健康信息参考，不构成医学诊断或治疗建议。** 系统在所有AI回答末尾均标注以下声明：
> "AI生成内容仅供参考，不构成诊断建议。如有身体不适，请及时就医。"

### 4. 隐私与安全

- 用户密码使用 bcrypt 哈希加密存储
- JWT 双 Token 机制（Access Token 15min + Refresh Token 7d）
- 所有用户上传的图片和文件仅用于本次会话分析，不持久存储
- API 请求频率限制：100次/15分钟/IP
- XSS 防护：所有 AI 生成的 Markdown 内容经 DOMPurify 过滤后渲染
- CORS 白名单 + Helmet 安全头

### 5. 关于联邦学习模块

系统包含联邦学习原型实现，模拟跨医院的隐私保护联合搜索。该模块展示了以下能力：
- 差分隐私预算管理
- 医院节点注册与公钥加密
- 跨节点安全查询
- 操作审计日志

该模块为技术演示性质，展示了本系统在医疗数据隐私保护方面的探索。

### 6. 项目开源说明

本项目基于 MIT 协议开源，所有前端代码（Vue 3 + VTK.js）、后端代码（Fastify + TypeScript）、AI 管道编排代码均完全公开。欢迎学术界和开发者社区参与贡献。
