## 三、AI在作品中的应用说明

### 3.1 智能分诊与多模型路由

系统内置智能路由引擎，根据用户问题的复杂度自动选择模型：

- **简单问题**（如"感冒了吃什么"）→ 路由至 `qwen3.5-flash`，快速响应
- **复杂问题**（如"体检报告多项指标异常分析"）→ 路由至 `qwen-max`，深度分析
- 用户也可手动选择"智能选择""深度分析""快速回答"三种模式

### 3.2 检索增强生成（RAG）可信问答

为解决大语言模型在医疗场景中的"幻觉"问题，系统实现了完整的RAG管线：

1. **知识库构建**：将权威医学文献、临床指南分块（chunk）并向量化存储
2. **语义检索**：用户提问时，将问题向量化后在pgvector中执行相似度搜索
3. **重排序**：使用跨编码器（cross-encoder）对检索结果重排序，提升精度
4. **多源融合**：融合向量检索 + 知识图谱查询结果，输入LLM生成回答
5. **可解释输出**：每条回答附带来源引用，用户可追溯依据

### 3.3 医学图片智能识别

- **药品识别**：用户拍摄药品包装，通过视觉模型识别药品名称、成分、适应症、禁忌等信息
- **报告解读**：用户上传体检报告照片，自动OCR识别并结构化解析各项指标，标注异常值并给出科普性解读

### 3.4 多智能体系统（Multi-Agent）

系统内置多个专业化AI智能体（Agent），根据用户输入自动匹配合适的专家角色：

| 智能体      | 专业领域      | 匹配关键词示例     |
| -------- | --------- | ----------- |
| 神经内科头痛专家 | 头痛、偏头痛、眩晕 | 头痛、偏头痛、头晕   |
| 中医专家     | 中医药、养生调理  | 中医、中药、针灸、体质 |
| 全科健康助手   | 常见病、体检解读  | 感冒、发烧、体检、指标 |
| 数据库优化师   | SQL、性能优化  | SQL、索引、慢查询  |
| 后端架构师    | 后端开发      | API、后端、接口   |
| 前端开发者    | 前端开发      | Vue、组件、CSS  |

### 3.5 知识图谱增强推理

系统维护医学知识图谱，包含疾病、症状、药品、检查等实体及其关系。在回答问题时：

1. 解析用户问题中的医学实体
2. 从知识图谱中检索相关实体及关系路径
3. 将图谱信息与向量检索结果融合
4. 输入LLM生成有依据的回答

### 3.6 AI伦理与安全设计

系统提示词（System Prompt）包含完整的可信性约束：

- **可信性铁律**：每条健康建议必须附带来源标注
- **交互性铁律**：信息不足时优先追问而非武断回答
- **安全性铁律**：识别到紧急情况（胸痛、呼吸困难等）时优先提示就医
- **禁止行为**：不得诊断疾病、不得开具处方、不得给出具体剂量

## 四、作品简介

### 4.1 作品概述

"智疗助手"是一个面向大众的AI健康咨询平台，旨在通过人工智能技术为用户提供可信、专业、可追溯的医疗健康信息服务。作品采用前后端分离架构，前端基于Vue 3构建现代化响应式界面，后端基于Node.js + Fastify构建高性能API服务，以Docker容器化方式一键部署。

### 4.2 核心功能

| 功能模块           | 说明                                    |
| -------------- | ------------------------------------- |
| **智能健康问答**     | 基于通义千问大模型的AI对话，支持多模型智能路由              |
| **深度医学搜索**     | 结合RAG技术检索权威医学文献库，返回有据可查的回答            |
| **体检报告解读**     | 上传体检报告图片，AI自动识别并解读各项指标                |
| **药品识别**       | 拍照识别药品包装盒，获取药品详细信息                    |
| **肺部CT 3D可视化** | 支持DICOM格式CT数据加载与VTK.js三维体渲染，模拟肺结节数据预览 |
| **医学知识库管理**    | 支持自定义知识文档的增删改查与语义搜索                   |
| **语音输入**       | 基于Web Speech API的中文语音识别               |
| **对话管理**       | 多会话管理、Markdown导出                      |
| **用户系统**       | 手机号注册/登录、JWT认证、个人资料管理                 |

## 七、设计思路

### 7.1 设计背景与动机

随着医疗资源分布不均问题的日益突出，大众对便捷、可信的健康信息服务需求急剧增长。然而，现有AI健康助手普遍存在以下问题：

1. **"幻觉"问题**：通用大模型在医疗领域易生成看似合理但实际错误的信息
2. **可信度不足**：回答缺乏来源引用，用户无法验证信息真实性
3. **交互方式单一**：仅支持纯文本交互，缺乏对医学影像、药物图片等非文本信息的理解
4. **隐私顾虑**：用户健康数据敏感，需要严格的数据保护机制

基于以上痛点，本作品设计了"智疗助手"——一个以 **可信、安全、多模态** 为核心设计理念的AI健康咨询平台。

### 7.2 整体架构设计思路

**分层解耦架构**：系统采用前后端分离、微服务化的架构设计。前端专注用户体验与交互，后端专注业务逻辑与AI推理，通过RESTful API + SSE（Server-Sent Events）实现流式通信。

**容器化部署**：所有组件（Web、Server、PostgreSQL、Redis）通过Docker Compose统一编排，实现"一键部署"，降低部署门槛。

**插件化AI设计**：AI能力模块（LLM调用、RAG检索、知识图谱推理、联邦学习等）作为独立服务注入，便于后续扩展和替换。

### 7.3 可信AI设计思路

本作品最核心的设计思路是**解决AI医疗回答的可信度问题**，采用了三级保障体系：

**第一级：RAG检索增强生成**

- 构建医学知识库（权威指南、文献、教材）并向量化存储
- 用户提问时，先从知识库检索相关文档片段
- 将检索结果作为上下文注入LLM，约束回答范围
- 每条回答标注来源出处，支持用户溯源验证

**第二级：知识图谱增强推理**

- 构建疾病-症状-药品-检查实体关系图
- 对用户问题进行实体识别和关系推理
- 融合图谱路径信息，增强回答的逻辑完整性

**第三级：可解释AI输出**

- 对每条回答生成置信度评分
- 展示推理链：用户问题 → 检索来源 → 推理过程 → 最终回答
- 明确标注确定性信息与不确定信息

### 7.4 交互设计思路

**多模态交互**：

- 文本对话：核心交互方式，支持流式打字输出
- 图片输入：药品拍照识别、报告拍照解读
- 语音输入：基于Web Speech API的中文语音识别
- 3D可视化：CT影像的交互式体渲染

**渐进式功能引导**：

- 首页展示五个核心功能入口（健康问答、深度搜索、报告解读、药盒识别、肺部CT）
- 用户根据需求选择功能，系统自动切换到对应模式
- 对话中支持模式切换，无需退出当前会话

### 7.5 隐私安全设计

- JWT双Token认证机制（Access Token + Refresh Token）
- 密码bcrypt哈希存储
- 支持短信验证码登录/注册（阿里云SMS）
- CORS跨域安全策略
- Helmet安全头防护
- Rate Limiter限流保护
- 联邦学习差分隐私保护

## 八、设计重点难点

### 8.1 重点一：医疗RAG系统的构建与优化

**难点**：

- 医疗文本专业性强，通用分词和嵌入模型效果不佳
- 检索结果需要高精度，错误信息可能造成严重后果
- 知识库数据量大，实时检索性能要求高

**解决方案**：

- 使用BGE-M3嵌入模型（专为多语言和领域文本优化），生成1024维向量
- 双层检索策略：向量相似度检索（初筛）→ Cross-Encoder重排序（精排）
- 混合搜索：结合向量检索 + 全文关键词检索 + 知识图谱查询
- pgvector IVFFlat索引加速近似最近邻搜索
- 分块策略：512字符分块+128字符重叠，保留上下文完整性
- 相似度阈值过滤（0.7），低于阈值的检索结果不送入LLM

### 8.2 重点二：3D医学影像体渲染

**难点**：

- 医疗DICOM数据格式复杂，解析门槛高
- 三维体渲染计算量大，浏览器端性能受限
- 需要模拟肺结节数据用于演示和测试

**解决方案**：

- 集成VTK.js高性能体渲染管线
- 支持DICOM标准格式解析（dicom-parser）
- 集成Cornerstone3D医学影像渲染引擎
- 模拟CT数据生成器：程序化生成含肺结节、血管结构的模拟CT数据
- GPU加速体绘制，支持实时窗宽窗位调节
- 多平面重建（MPR）支持

### 8.3 重点三：多模态AI融合

**难点**：

- 文本、图像、3D数据需要统一的理解框架
- 不同模态的数据处理流程差异大
- 视觉模型对医学图片的识别精度要求高

**解决方案**：

- 统一API网关层，不同模态走不同处理管线
- 文本：LLM + RAG管线
- 图片：视觉模型（qwen3-vl-plus）识别 + LLM解读
- 3D：VTK.js渲染 + 分析结果融合
- 使用阿里云百炼的统一API接口，降低多模型接入复杂度

### 8.4 重点四：联邦学习隐私保护

**难点**：

- 医疗数据高度敏感，不能直接共享
- 跨机构数据协作需要隐私保护机制
- 模型聚合需要保证效果不显著下降

**解决方案**：

- 实现联邦学习框架，各医院节点在本地计算模型更新
- 差分隐私（Differential Privacy）机制，ε参数可配置
- 安全聚合（Secure Aggregation），防止单点数据泄露
- 数据完整性验证（Hash校验），防篡改
- 隐私预算追踪，控制整体信息泄露风险

### 8.5 重点五：系统可靠性保证

**难点**：

- 对接外部API（阿里云百炼）存在网络不稳定风险
- 流式响应（SSE）需要维持长连接
- 医疗场景对系统可用性要求高

**解决方案**：

- 指数退避重试机制：3次自动重试，间隔指数增长（1s→2s→4s）
- 可重试状态码定义：408（超时）、429（限流）、5xx（服务端错误）
- SSE长连接超时配置（proxy\_read\_timeout: 300s）
- Redis缓存层，减少数据库压力
- Docker健康检查 + 自动重启策略
- 链路日志追踪（pino结构化日志）
